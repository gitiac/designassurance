Why cloud network design is still network design

9 April 2026  ·  Hameed Nazim, CCDE | CCIE  ·  7 min read

Cloud architecture is regularly discussed as if the network layer designs itself - a commodity handled by the hyperscaler that the architect does not need to think about too carefully. After reviewing cloud migration and transformation programmes across financial services, biopharmaceutical, healthcare and automotive environments, I can say with confidence that this assumption is one of the most consistent sources of post-migration failure.

The network is not a commodity. In complex enterprise environments, it is the architecture. Everything else - application performance, security posture, compliance, resilience - depends on the network design being correct.

The network is not a commodity. In complex enterprise environments, it is the architecture. Everything else - application performance, security posture, compliance, resilience - depends on the network design being correct. And getting it correct in a cloud environment requires the same depth of thinking that enterprise network design has always required.

What cloud network design actually involves

A cloud network design for an enterprise programme involves decisions that cannot be reversed cheaply once made. VPC and VNet architecture - how the network is segmented, which segments can communicate and which cannot, how those boundaries are enforced - shapes every security, compliance and connectivity decision that follows. Getting this wrong at the start means rebuilding it later, under pressure, with production systems in place.

Connectivity architecture - whether to use ExpressRoute, Direct Connect, Transit Gateway, VPN or a combination - determines the latency, bandwidth, resilience and cost profile of everything that connects to the cloud environment. These decisions interact with on-premises infrastructure, with security architecture, with regulatory requirements around data sovereignty, and with the application performance profiles that the business has promised its users.

Latency and MTU - the two numbers cloud architecture gets wrong most often

Latency is the number most cloud network designs either ignore or underestimate. On-premises applications are often designed with the assumption of sub-millisecond network latency within a data centre. When those applications move to cloud - or when they need to communicate between cloud and on-premises environments - the latency profile changes significantly. A design that does not model the expected latency under real traffic conditions, and validate that the application can tolerate it, is not a complete design.

MTU is equally critical and consistently overlooked. Maximum Transmission Unit mismatches between on-premises and cloud environments, or between different cloud services and connectivity paths, cause fragmentation, retransmission and degraded throughput that is notoriously difficult to diagnose once in production. ExpressRoute and VPN tunnels, overlay networks and encapsulation protocols all introduce MTU constraints that must be designed for explicitly. A network that works perfectly in testing - where traffic volumes are low and the impact of fragmentation is masked - can degrade significantly under production load when MTU issues surface at scale.

Both latency and MTU need to be modelled at design stage, validated against application requirements, and documented in the LLD with specific configuration values - not left as implementation details to be resolved later.

Security architecture in cloud environments

Cloud security architecture is not a firewall rule set applied at the perimeter. In a well-designed cloud network, security is enforced at multiple layers - at the VPC or VNet boundary, at the subnet level, at the endpoint level, and at the application layer - with each layer designed to be independently defensible.

The mistake I see most frequently is treating cloud security as equivalent to on-premises perimeter security, simply moved to a virtual appliance. The result is a flat, over-permissive network that passes a checkbox review but would not survive a lateral movement attack. In regulated environments - financial services, healthcare, biopharmaceutical - this is not just a design problem. It is a compliance problem that surfaces at audit.

Security groups, network access control lists, private endpoints, service endpoints and private DNS resolution all need to be designed together, not configured independently. The interactions between them are where the gaps appear.

The AWS Landing Zone - what the network design actually determines

AWS Landing Zone and Azure Landing Zone frameworks provide an opinionated starting point for cloud network architecture. They are not a substitute for network design. A Landing Zone tells you how to organise your accounts and management hierarchy. It does not tell you how to design your VPC CIDR ranges, your Transit Gateway routing tables, your Direct Connect or ExpressRoute redundancy model, or your hybrid DNS architecture.

A Landing Zone tells you how to organise your accounts and management hierarchy. It does not tell you how to design your network - and those are not the same thing.

I have reviewed programmes where a Landing Zone was deployed and the team considered the network design done. The Landing Zone was correctly deployed. The network design - the decisions that determine whether the environment will actually work for the organisation's specific applications, traffic patterns, security requirements and compliance obligations - had not been done at all.

What independent cloud network review looks for

When I review a cloud network design, I am looking for the decisions that will determine whether the environment works in production - not just in a proof of concept or a testing environment. The questions I ask are the same questions a CCDE-level network architect asks of any design:

Does the segmentation model reflect the actual security and compliance requirements, or is it a simplified approximation that will need to be reworked when a real requirement is applied to it? Has the latency between cloud and on-premises been modelled against the applications that will use it, and have MTU constraints been identified and designed for? Is the routing architecture resilient, or does it have a single point of failure that the redundancy diagrams do not reveal? Does the LLD specify actual configuration values - IP ranges, ASNs, BGP communities, MTU settings, security group rules - or does it describe intent without specifying implementation?

Cloud network design is still network design. The platform is different. The principles are not.

What I consistently find in cloud network design review
  • Latency and MTU have not been modelled against application requirements. The design assumes performance will be acceptable. Production reveals it is not.
  • The Landing Zone is treated as the network design. Account structure and management hierarchy are in place. Routing, segmentation, MTU and hybrid connectivity design are not.
  • Security is perimeter-based rather than layered. A virtual firewall appliance at the edge. Flat, over-permissive networks behind it.
  • The LLD contains intent, not configuration. "BGP will be used" rather than the ASNs, communities, route filters and failover behaviour that make BGP work correctly in this specific environment.
  • Hybrid connectivity resilience is designed at the circuit level, not the routing level. Two ExpressRoute circuits. One BGP routing design that fails over to neither of them correctly under real failure conditions.
